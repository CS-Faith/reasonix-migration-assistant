# 扒开源码，手写脚本：我把 Reasonix 0.53 的一百轮对话，搬进了新版

> 这不是一篇"双击运行即可"的教程。这是一篇关于逆向数据格式、踩遍 JSON 编码陷阱、最终写出 346 行 PowerShell 脚本的全记录。

---

## 一、问题

Reasonix 从 0.53 到 1.x，数据目录结构完全变了。

老版本 `.reasonix/sessions/` 下面是这样：

```
desktop-202605270312-1.jsonl       ← 对话内容
desktop-202605270312-1.meta.json   ← 元数据（标题/token/费用）
```

新版 `%APPDATA%/reasonix/projects/<slug>/sessions/` 下面是这样：

```
20260527-031200.000000000-deepseek-v4-flash.jsonl             ← 对话内容
20260527-031200.000000000-deepseek-v4-flash.jsonl.meta        ← 分支元数据
20260527-031200.000000000-deepseek-v4-flash.jsonl.telemetry.json ← 运行指标
20260527-031200.000000000-deepseek-v4-flash.ckpt/             ← 检查点
```

文件名变了。后缀变了。一套变成了四件套。数据直接丢进去，新版完全不认。

唯一的办法：**读新版 Go 源码，反推它到底要什么。**

## 二、反推数据格式

### 2.1 分支元数据 (BranchMeta)

在 GitHub 上找到 `branch.go`，BranchMeta 结构体长这样：

```go
type BranchMeta struct {
    ID            string    `json:"id"`
    CreatedAt     time.Time `json:"created_at"`
    UpdatedAt     time.Time `json:"updated_at"`
    Scope         string    `json:"scope,omitempty"`
    WorkspaceRoot string    `json:"workspace_root,omitempty"`
    TopicID       string    `json:"topic_id,omitempty"`
    TopicTitle    string    `json:"topic_title,omitempty"`
}
```

几个关键信息，全是踩坑踩出来的：

**snake_case，不是 camelCase。** 最初我按 camelCase 写 `topicId`、`workspaceRoot`，新版完全不理。必须从 struct tag 里读字段名，不能猜。

**时间用 RFC3339。** `2026-06-04T07:41:03Z`，精确到秒即可。多个 Z 少个 Z 无所谓，但格式必须对。

**omitempty 的意思是：空字符串可以省略。** 也就是说 `topic_id: ""` 和 `topic_id` 不存在，Go 解析出来都是空字符串。但我们后来发现，要不要留空，直接影响侧边栏的行为（后面会讲）。

### 2.2 运行指标 (telemetry)

新版在每个会话旁边存一份 `telemetry.json`：

```json
{
  "version": 2,
  "readFiles": [{"path": "...", "turn": 4, "time": 0, "offset": 0, "limit": 50}],
  "usage": {
    "promptTokens": 168314,
    "completionTokens": 68109,
    "totalTokens": 236423,
    "cacheHitTokens": 12091392,
    "cacheMissTokens": 200578,
    "requestCount": 1,
    "elapsedMs": 1,
    "sessionCost": 0.19,
    "sessionCurrency": "CNY"
  }
}
```

老版的指标分散在 `.meta.json` 里，字段名不同：

| 老版字段 | 新版字段 | 含义 |
|---------|---------|------|
| `lastPromptTokens` | `promptTokens` | 最后一次请求的 prompt token |
| `totalCompletionTokens` | `completionTokens` | 总计输出 token |
| `cacheHitTokens` | `cacheHitTokens` | 缓存命中 token |
| `cacheMissTokens` | `cacheMissTokens` | 缓存未命中 token |
| `totalCostUsd` | `sessionCost` / `sessionCostUsd` | 费用 |

**这里有一个魔鬼细节。** `requestCount` 字段不能写 0。

我在新版源码 `tabs.go` 里发现，`loadTelemetry()` 函数有一个判断：

```go
if len(snapshot.ReadFiles) > 0 || snapshot.Usage.RequestCount > 0 {
    // 加载 telemetry 数据
}
```

注意是 `>` 不是 `>=`。如果 `requestCount` 为 0，telemetry **完全不加载**。所有指标——token、缓存命中率、费用——全部显示为 0。填成 1，一切正常。

这就是读源码的意义：不看代码，这个 bug 永远找不到。

### 2.3 依赖文件 (readFiles)

新版的 telemetry 里有一个 `readFiles` 数组，记录会话中读了哪些文件。

老版没有单独存这个数据。但老版的 jsonl 里，每一行是一条消息。工具调用记录长这样：

```json
{
  "role": "assistant",
  "tool_calls": [{
    "id": "call_xxx",
    "type": "function",
    "function": {
      "name": "read_file",
      "arguments": "{\"path\":\"C:/path/to/file.md\",\"offset\":0,\"limit\":100}"
    }
  }]
}
```

遍历老版 jsonl，解析 `read_file` 调用，提取 `path`、`turn`、`offset`、`limit`，组装成新版格式。一共从老版会话里提取了 190 条依赖文件记录。

### 2.4 检查点 (ckpt)

新版每个会话有一个 `.ckpt/` 文件夹，里面是 `turn-0.json`、`turn-1.json`...

格式很简单：

```json
{"turn": 0, "time": "2026-06-04T07:41:03Z", "prompt": "用户第一句话（截取前500字）", "msgIndex": 0, "files": null}
```

遍历 jsonl，找到每个 `role: "user"` 的消息，生成对应的 turn-N.json 即可。

## 三、MCP 配置：从 JSON 到 TOML

老版 MCP 配置在 `config.json` 里，格式是 JSON：

```json
{
  "mcp": {
    "image-analyzer": {
      "command": "C:/Users/xxx/python.exe",
      "args": ["C:/Users/xxx/image_mcp_server.py"]
    }
  }
}
```

新版用 TOML 的 `[[plugins]]`：

```toml
[[plugins]]
name = "image-analyzer"
command = 'C:/Users/xxx/python.exe'
args = ['C:/Users/xxx/image_mcp_server.py']
```

注意 Windows 路径用**单引号**包裹。如果用了双引号，TOML 的 `\` 会被当作转义字符，路径直接坏掉。

## 四、最惨烈的坑：BOM 问题

这一点值得单独写一节。

**症状：** 历史对话列表能看到迁移的会话，标题正确，轮数正确，但点"打开会话"永远报错——"无法打开会话"。

**排查过程：** 我反复检查了文件名、meta 格式、telemetry 内容、topic_id 设置。一切看起来都对。新版为什么就是打不开？

**根因定位：** 新版 Go 源码里，加载会话的函数 `findTopicSession()` 调用 `LoadBranchMeta()`。`LoadBranchMeta()` 用 Go 标准库的 `json.Unmarshal()` 解析 meta 文件。

Go 的 `json.Unmarshal` **不跳过 BOM（Byte Order Mark，字节序标记）**。

而 PowerShell 的 `Set-Content -Encoding UTF8` 默认在文件开头写入 BOM：`0xEF 0xBB 0xBF`。

所以 meta 文件前三个字节是 BOM，Go 解析 JSON 时遇到非法字符，直接失败。`findTopicSession()` 没找到会话 → 新建一个空会话 → 用户看到"无法打开会话"。

**修了 123 个 meta 文件才定位到这个 bug。**

**修复：** 全程改用 `[System.Text.UTF8Encoding]::new($false)`（无 BOM 编码）+ `[System.IO.File]::WriteAllText` 写入。

## 五、侧边栏为什么会被刷屏

另一个折磨人的问题：迁移后打开新版，左侧边栏冒出上百条对话记录，全是 `topic_20260527-031200-xxx` 这种自动生成的名字。

**根因：** 新版源码 `tabs.go` 第 1519 行，`orderedTopicIDs()` 函数：

```go
func orderedTopicIDs(explicit []string, titleMap map[string]string) []string {
    seen := map[string]bool{}
    out := make([]string, 0, len(explicit)+len(titleMap))
    for _, tid := range explicit {
        // ... 先加入显式列表
    }
    for tid := range titleMap {    // ← 这里！遍历 titleMap 所有 key！
        if !seen[tid] {
            out = append(out, tid)
        }
    }
    return out
}
```

这个函数不仅把 `desktop-projects.json` 里 `globalTopics` 列表中的话题渲染到侧边栏，**还把 `desktop-topic-titles.json` 里所有 key 都加进去**。

即使 `globalTopics` 为空，只要 `titleMap` 有 128 个条目，侧边栏就显示 128 条。

**解决：** 迁移前清空 `desktop-topic-titles.json` 和 `desktop-topic-title-sources.json`。同时在根 `sessions/` 目录下放置 `.legacy-imported.v2-routed` 标记文件，阻止新版启动时自动运行迁移扫描。

这个发现花了三个小时。读源码之前，我一直以为是 `desktop-projects.json` 的问题——直到逐行跟踪 `ListProjectTree()` 的执行流程才发现真相。

## 六、架构设计

整个工具只有一个 PowerShell 脚本（346 行）和一个 BAT 引导文件。

```
Migrate-053to1X.ps1
  ├── 源检测（C盘扫描 / 便携目录自定位）
  ├── [1/4] MCP 迁移
  ├── [2/4] 会话迁移（重命名 + 四件套生成）
  ├── [3/4] 记忆迁移
  └── [4/4] 历史发现（topic_id + 标题 + 标记 + 状态清理）
```

两个场景（标准升级 / 便携版升级）只在第一步源检测处分叉，后续四步逻辑完全共享。这样维护时只需改一处，不会出现"标准版修了但便携版忘了"的情况。

## 七、总结

| 难题 | 根因 | 解决方案 |
|------|------|---------|
| meta 字段名不匹配 | Go struct tag 是什么就是什么 | 抄源码 |
| telemetry 不显示 | `requestCount: 0` 不触发加载 | `requestCount: 1` |
| 会话打不开 | PowerShell BOM vs Go json.Unmarshal | 无 BOM 写入 |
| 侧边栏被刷屏 | `orderedTopicIDs` 遍历 titleMap 所有 key | 清空 title 文件 + 放迁移标记 |
| 对话标题都是"你好" | 标题文件缺失覆盖了 meta 里的值 | 保留老版 `summary` 字段 |

**核心经验：不要猜。读源码。**

新版 Reasonix 是 Go 写的，开源在 GitHub。从 `branch.go` 的 struct tag，到 `tabs.go` 的 `orderedTopicIDs` 逻辑，到 `save.go` 的 `loadTelemetry` 判断——每一个行为都有据可查。

346 行脚本的背后，是无数次的 `curl https://raw.githubusercontent.com/...`，是对着 Go 源码逐行逆向，是修了 123 个文件才定位到的 BOM bug。

开源地址：https://github.com/CS-Faith/reasonix-migration-assistant
