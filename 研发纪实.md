# Reasonix 配置迁移升级助手 — 研发纪实

## 第一阶段：摸着石头过河（理解 1.5 数据格式）

### 坑 #1：文件名格式不匹配

0.53 命名 `desktop-YYYYMMDDHHMM-N.jsonl`，1.5 命名 `YYYYMMDD-HHMM00.000000000-deepseek-v4-flash.jsonl`。时间戳提取时要注意 `N` 后缀的处理——0.53 允许同一分钟多次对话。

**教训**：先用 DryRun 模式批量测试命名规则，确保没有碰撞。

### 坑 #2：BranchMeta 字段名混乱

1.5 的 `BranchMeta` 使用 **snake_case** JSON（`topic_id`, `workspace_root`, `created_at`），不是 camelCase。最初我们按 camelCase 写入，1.5 完全不认。

**教训**：必须从 Go 源码 struct 的 `json` tag 确认字段名，不能靠猜。

### 坑 #3：meta 文件名格式

0.53 的 meta 叫 `.meta.json`，1.5 的 meta 叫 `.jsonl.meta`（`BranchMetaPath = sessionPath + ".meta"`）。

**教训**：文件后缀链式拼接，`.jsonl` 后面再加 `.meta`。

## 第二阶段：反复折腾（会话不显示）

### 坑 #4：ConvertTo-Json 双反斜杠

PowerShell 的 `ConvertTo-Json` 会把 `\` 转义为 `\\`，这恰好是 JSON 规范要求的。但我们在拼接 `desktop-tabs.json` 时额外做了一次 `-replace '\\','\\'`，把正确的双反斜杠变成了四反斜杠。

**教训**：`ConvertTo-Json` 的输出就是正确的 JSON，不需要再手动转义反斜杠。如果要手写 JSON 字符串，用 `@""@` here-string 配合 `-replace` 比 `ConvertTo-Json` 更可控。

### 坑 #5：日常覆盖战

1.5 启动时读取 `desktop-tabs.json`，关闭时写回。如果你在 1.5 关闭时修改了这些文件，1.5 下次启动会**覆盖**你的修改。`desktop-projects.json` 在 `saveProjectsFile()` 被调用时写入，触发时机包括：
- `EnsureBlankTab()` 创建话题
- `migrateLegacySessionsIntoGlobalTopics()` 注册迁移话题
- `OpenGlobalTab()` 打开新话题

**教训**：不要试图在 1.5 运行时或刚关闭时修改 JSON 文件——1.5 内部有互相引用的状态，外部修改必然被覆盖。正确的做法是：让 1.5 自己发现并管理状态，或完全替换所有关联文件。

### 坑 #6：侧边栏幽灵条目

`orderedTopicIDs()` 不仅从 `desktop-projects.json.globalTopics` 取话题，还从 `desktop-topic-titles.json` 里取**所有 key**。即使 `globalTopics` 为空，titleMap 里的 128 个条目全被渲染到侧边栏。

**教训**：读源码！这个 bug 折腾了好几个小时，最后是在 `tabs.go` 第 1519 行找到的：
```go
for tid := range titleMap {
    if !seen[tid] { out = append(out, tid) }
}
```

### 坑 #7：loadTelemetry 的 requestCount 门槛

`loadTelemetry()` 只有满足条件才加载数据：
```go
if len(snapshot.ReadFiles) > 0 || snapshot.Usage.RequestCount > 0 {
```

我们发现 `requestCount: 0` 时，telemetry 数据**完全不加载**——token 数、缓存命中率、费用等全部显示为 0。设为 1 后正常加载。

**教训**：看代码时要注意 `>` 而非 `>=`。

## 第三阶段：BOM 之痛（会话打不开）

### 坑 #8：PowerShell UTF-8 BOM

症状：历史对话列表能看到，但"打开会话"永远失败。

PowerShell 的 `Set-Content -Encoding UTF8` 在文件开头写入 **BOM（0xEF BB BF）**。Go 的 `json.Unmarshal()` 不跳过 BOM，直接解析失败 → `LoadBranchMeta()` 返回错误 → `findTopicSession()` 跳过该会话 → 新建空会话 → "无法打开会话"。

修了 123 个 meta 文件才定位到。

**教训**：跨语言文件交换时，永远使用 `new UTF8Encoding(false)`（无 BOM）。PowerShell 中对应的写法是：
```powershell
$enc = [System.Text.UTF8Encoding]::new($false)
[System.IO.File]::WriteAllText($path, $content, $enc)
```

## 第四阶段：精雕细琢

### 进展 #1：readFiles 提取

从 0.53 jsonl 的工具调用记录中解析 `read_file` 调用，提取路径、turn、offset/limit，填充到 1.5 的 telemetry.readFiles 数组。

### 进展 #2：对话标题保留

从 0.53 的 `.meta.json` 提取 `summary` 字段，写入 1.5 的 `topic_title`。这样历史对话列表里每条记录都保留了原始标题，而不是显示文件名或"你好"。

### 进展 #3：迁移标记绕过侧边栏污染

在根 `sessions\` 目录放置 `.legacy-imported.v2-routed` 标记文件，阻止 `migrateLegacySessionsIntoGlobalTopics()` 运行，从而避免侧边栏被旧会话刷屏。会话通过 `ListBranches()` 直接出现在历史对话列表中。

## 总结：最有价值的教训

| 编号 | 坑 | 根因 | 耗时 |
|------|------|------|------|
| #1 | 文件名碰撞 | 不了解命名规则 | 30min |
| #2 | meta 字段名 | 没读 Go struct tag | 1h |
| #5 | JSON 被覆盖 | 不知道 1.5 写回时机 | 2h |
| #6 | 侧边栏幽灵条目 | orderedTopicIDs 的 titleMap 循环 | 3h |
| #8 | 会话打不开 | PowerShell BOM vs Go json.Unmarshal | 2h |

**核心原则**：不猜——读源码。新版的所有行为都能在 GitHub 的 Go 源码中找到答案。
